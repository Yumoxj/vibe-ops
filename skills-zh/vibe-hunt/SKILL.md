# Vibe Hunt

## Overview

诊断驱动的调试技能。只针对症状打补丁会在别处制造新 bug。

**在能用一句话陈述根因之前，不要碰代码：**
> "根因是 [X]，证据是 [Y]。"

必须指出具体的文件、函数、行号或条件。"一个状态管理问题"不可验证。"`src/hooks/user.ts:42` 的 `useUser` 缓存过期，因为依赖数组缺少 `userId`"可验证。说不到这么具体，说明还没有假设。

---

## When to Use

**使用场景：**
- Bug、错误、崩溃、异常行为
- 测试失败
- 开发卡壳、多次尝试无进展

**不适用场景：**
- 代码审查（使用 /vibe-review）
- 新功能开发
- 设计讨论

---

## 理性化监控

这些想法出现时，停下来重新审视：

| 你在想什么 | 实际意味着 | 规则 |
|---|---|---|
| "我先试一下这个" | 没有假设，随机游走 | 停。先写假设。 |
| "我确定是 X" | 信心不是证据 | 跑一个能证明它的工具。 |
| "可能和之前的问题一样" | 把新症状当成已知模式 | 从头重新读执行路径。 |
| "我机器上没问题" | 环境差异就是 bug | 列举每个环境差异再判断。 |
| "再重启一次应该就好了" | 在回避错误信息 | 逐字读最后的错误。重启不超过两次且无新证据。 |
| "终于出现不同的错误了！" | 调试 1h 后的随机修改产生新症状是巧合，不是进展 | 如果已达到 Level 3 阈值，无论最近的任何变化，保存状态并重新开始。 |

---

## 进展信号

这些出现时，诊断方向正确：

| 你在想什么 | 意味着 | 下一步 |
|---|---|---|
| "这行日志和假设吻合" | 找到正向证据 | 再找一个独立证据交叉验证 |
| "我能预测下一个错误" | 心智模型正在形成 | 跑预测；匹配则模型正确 |
| "根因在 A 但症状出现在 B" | 传播路径已理解 | 追踪 A→B 调用链，确认每一环 |
| "我能写一个在旧代码上失败的测试" | 假设具体且可测试 | 修复前先写测试 |

没有可观察证据匹配以上任一信号，不能声称有进展。

---

## 硬规则

- **修复前必须报告发现。** 输出诊断报告（根因、受影响文件、修复方案），等用户确认后才能修改代码。
- **修复后症状相同 = 硬停。** 上一个假设是错的。从头重新读执行路径。
- **三个假设失败后，停下。** 向用户汇报：检查了什么、排除了什么、未知什么，询问如何继续。
- **环境细节不从记忆中说。** 先跑检测命令（`node --version`、`rustc --version` 等），陈述实际输出。
- **外部工具失败：先诊断再切换。** MCP 工具或 API 失败时，先确定原因（服务运行？密钥有效？配置正确？），再尝试替代方案。
- **注意回避信号。** 当某人说"那部分不重要"，当成线索。回避检查的区域往往是问题所在。
- **修复原因，不修复症状。** 如果修复涉及 5 个以上文件，暂停并与用户确认范围。
- **碰代码前先 Grep。** 修改任何文件或调用任何标识符前，grep 确认它存在于预期位置。无结果 = 重新检查执行路径，不要凭记忆编辑。

## 卡壳升级策略

当常规诊断流程受阻时，逐步升级：

### Level 1：假设失败（单一角度 15 分钟以上）
- 同一症状修复后仍存在 → 从头重新读执行路径
- 切换分析角度：日志、调用链、状态追踪

### Level 2：3 次假设失败（累计 30 分钟以上）
- 执行 `/rewind` 回退会话
- 深度分析：代码结构、工作原理、关键组件
- 向用户汇报已知/已排除/未知

### Level 3：严重卡壳（累计调试 1 小时以上）
1 小时阈值是绝对的——衡量的是总调试时间，而非距上次"信号"的时间。调试 1 小时后，你的心智模型已经不可靠。一系列随机修改后出现的"新错误"是巧合变异，不是进展。

- 先执行 `git stash` 暂存当前修改
- 使用 AskUserQuestion 确认是否 `git reset --hard`（提示：stash 已保存修改）
- 新建 `/new` 会话
- 全局分析：架构、模块划分、连锁反应评估
- 从零开始建立心智模型

## 验证或证伪

添加一个针对性检测工具：一条日志、一个失败断言、或一个如果假设正确就会失败的最小测试。运行它。证据与假设矛盾 → 完全证伪假设，用刚学到的重新定位。不要保留被证据证伪的假设。

---

## 诊断报告（修改代码前必须输出）

确认根因后，输出以下报告并停止：

```
Root cause:    [什么错了, file:line]
Evidence:      [证明这是根因的证据]
Affected:      [需要修改的文件]
Proposed fix:  [修改方案及原因]
Risk:          [修复可能影响的其他部分]
```

**停。** 等用户确认后才能执行修改。

用户可能：批准修复、调整范围、要求继续诊断、或提出替代方案。

---

## 修复结果（修复完成后输出）

```
Root cause:  [什么错了, file:line]
Fix:         [改了什么, file:line]
Confirmed:   [证据或证明修复的测试]
Tests:       [pass/fail 数量, 回归测试位置]
```

状态：**resolved** / **resolved with caveats**（说明） / **blocked**（说明未知项）。

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 没假设就改代码 | 引入新 bug | 先陈述假设再动手 |
| 修复后症状依旧 | 越改越乱 | 停下来重读执行路径 |
| 凭记忆说版本号 | 诊断方向错误 | 跑检测命令，看实际输出 |
| 多次重启不看错误信息 | 浪费时间 | 逐字读最后的错误信息；重启不超过两次且无新证据 |
