# 专家审查员激活目录

编排器阅读完整 diff 并通过判断（非关键词匹配）决定激活哪些专家。本目录定义需要推理的信号。

## 始终开启（无需条件）

vibe-review Phase 3 基础审查始终运行（目标对齐、完整性、一致性、硬停止项）。专家审查员在其之上叠加。

## 条件专家

### 安全审查员

**Agent 文件：** `agents/reviewer-security.md`
**激活级别：** Standard 或 Deep 深度

diff 涉及以下内容时激活：
- 认证或授权逻辑（中间件、守卫、JWT 处理、会话管理）
- 加密操作（哈希、签名、加密）
- 信任边界的输入处理（表单字段、API 请求体、URL 参数）
- 用户可控路径的文件系统操作
- Shell 或子进程执行
- 第三方凭证或 API 密钥处理
- SQL 查询或原始数据库访问

**不激活：** 纯 UI 变更、配置文件更新、仅测试变更、文档。

### 架构审查员

**Agent 文件：** `agents/reviewer-architecture.md`
**激活级别：** Standard 或 Deep 深度

diff 涉及以下内容时激活：
- 添加新模块、包或服务边界
- 变更公共 API、导出类型或函数签名
- 引入此前不存在的跨模块导入
- 修改超过 10 个文件且分布在不同目录
- 添加或移除主要依赖
- 重构组件间的调用方式

**不激活：** 单文件 bug 修复、测试添加、样式变更、文档更新。

## 对抗性分析（仅 Deep）

无独立 agent。编排器在收集所有发现后将其作为额外推理轮次运行。

**激活级别：** 仅 Deep 深度（500+ 行变更，或明确高风险信号：认证、支付、数据变更、外部 API 集成）。

对抗性分析提问："如果我要通过这个 diff 攻击系统，会怎么做？"

四个攻击角度：
1. **假设违反** — 这段代码假设什么始终为真？（格式、顺序、范围）不为真时会发生什么？
2. **组合失败** — 新代码与现有系统在并发负载或部分失败下交互时会出什么问题？
3. **级联构造** — 什么有效的操作序列会导致无效状态？
4. **滥用场景** — 第 1000 次请求时、部署期间、两个用户同时编辑同一资源时会怎样？

对抗性发现附带置信度分数。抑制低于 0.60 的发现。
